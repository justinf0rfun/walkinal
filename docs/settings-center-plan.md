# 设置中心重构方案

## 1. 目标

将 Walkinal 当前的轻量设置 popover 升级为可扩展、可承载后续配置能力的独立设置中心。

本方案解决三个核心问题：

1. 当前设置入口承载能力不足
2. 后续大量功能配置缺少稳定落点
3. 当前设置体验仍停留在“工具弹层”，不符合产品级桌面应用体验

本方案只定义产品与交互实现方向，不直接涉及代码落地。

## 2. 当前问题

当前设置入口的主要问题：

- 设置入口是顶栏中的小型 popover，信息承载空间很有限
- 当前设置项之间缺少清晰的信息层级和分组结构
- 设置更像若干即时开关的集合，而不是完整设置中心
- 后续高价值配置项已经明确会继续增长，例如：
  - 快捷键自定义与冲突检测
  - Warp 跟随显示
  - 通知策略
  - 终端行为
  - 历史与瞬时指令策略
- 如果继续堆在现有 popover 中，复杂度会上升且体验快速恶化

## 3. 设计原则

### 3.1 设置中心是基础设施，不是附属弹层

设置中心需要成为承载未来配置项的稳定结构，而不是一次性的美化工程。

### 3.2 高频轻操作与低频深配置分离

不是所有设置都应该进入独立设置中心。

建议原则：

- 高频即时切换，可保留少量在主界面附近
- 涉及结构化选择、说明文案、冲突状态、组合配置的项，应进入独立设置中心

### 3.3 面向扩展而不是只服务当前 4 个开关

设置中心第一版就必须为未来 6-12 类配置预留稳定的信息架构。

### 3.4 不打断主工作流

设置中心需要独立，但不能让用户感觉“离开了主产品”。

## 4. 产品定位

建议将设置重构为：

**独立设置面板**

而不是继续扩展现有小 popover。

第一版建议形态：

- 使用独立设置窗口
- 从主悬浮层进入设置窗口
- 设置窗口具备独立布局、独立滚动和完整的信息密度
- 主悬浮层继续保持轻量工作流入口定位

这样做的原因：

- 未来设置项会持续增长，独立窗口更适合长期承载
- 设置中心不应继续挤占主悬浮层的有限空间
- 独立窗口更符合桌面级设置体验目标

## 5. 信息架构

建议设置中心采用左侧导航 + 右侧内容区结构。

第一版建议一级分组如下：

### 5.1 外观

包括：

- 主题模式
- 主界面宽度模式
- 视觉密度或未来可能的布局偏好

### 5.2 快捷键

包括：

- 主显示快捷键
- 冲突检测
- 快捷键状态反馈
- 后续可能增加的工作流快捷键

### 5.3 显示行为

包括：

- Warp 跟随显示
- 自动显示 / 自动隐藏
- 主界面显示策略
- 切换场景行为

### 5.4 通知

包括：

- 通知声音
- 发送完成反馈
- 失败反馈
- 后续通知策略扩展

### 5.5 存储

包括：

- Storage folder
- 历史与本地数据相关配置
- 后续数据管理能力

### 5.6 终端集成

包括：

- 当前终端目标
- 终端行为策略
- 后续 adapter 扩展入口

## 6. 交互形态

### 6.1 入口

入口仍然可以先保留在当前顶部三点按钮附近。

但点击后不再弹出小型 popover，而是进入完整设置中心。

### 6.2 退出

建议支持：

- 返回主界面
- Esc 关闭
- 顶部返回按钮或关闭按钮

### 6.3 保存策略

建议采用：

**即时保存**

适用于当前多数设置项，因为：

- 开关和选择器为主
- 用户期望调整后立即生效
- 不需要额外“保存”心智负担

但需要明确：

- 每项设置修改后要有轻微状态反馈
- 失败时要有可理解的错误提示

### 6.4 主界面与设置的关系

建议明确分工：

- 主悬浮层：负责输入、队列、发送、历史查看
- 设置窗口：负责结构化配置和低频高级设置

也就是说：

- `History` 保留在主界面
- 设置项迁入独立设置窗口

## 7. 视觉方向

目标不是“把当前 popover 放大”，而是形成桌面级设置体验。

建议视觉方向：

- 左侧窄导航栏
- 右侧内容主区域
- 每个分组有明确标题和辅助说明
- 开关、选择器、快捷键录入区域有稳定对齐和节奏
- 视觉层次要明显强于当前浮层小卡片

应达到的体验感受：

- 看起来像一个真正的产品设置中心
- 而不是临时工具面板

参考方向：

- 可参考 `Masko Code` 设置页的整体气质：
  - 左侧轻量导航
  - 右侧宽内容区
  - 分组式表单布局
  - 温和但明确的层级、边框和状态反馈

但 Walkinal 不应直接照搬，需要保留自身深色悬浮工具的气质。

## 7.1 UI 风格补充

基于参考图，建议第一版设置中心的 UI 风格补充如下：

### 侧栏

- 采用窄侧栏，承载一级导航
- 每个导航项使用图标 + 标题
- 当前选中项需要有清晰但不过度刺眼的高亮底色
- 侧栏底部可预留次级入口，例如反馈或文档链接

### 内容区

- 内容区采用分组块布局，而不是自由散点布局
- 每组配置应包含：
  - 分组标题
  - 简短说明
  - 配置控件
- 控件之间的间距要统一，避免当前 popover 那种“堆叠式开关列表”观感

### 控件风格

- 开关应有统一尺寸和对齐
- 分段选择器适合承载主题、模式切换等少量离散选项
- 快捷键录入区域需要独立视觉语言，不能只用普通按钮代替
- 路径类配置项应采用“信息 + 操作按钮”双栏结构

### 信息密度

- 比当前 popover 更高，但不能压迫
- 目标不是大而空，而是“桌面级可扫描”
- 每组内允许 2-5 个相关配置聚合，而不是一行只承载一个开关

## 7.2 交互细节补充

### 导航切换

- 左侧导航切换应即时响应
- 内容区切换要稳定，不应造成窗口尺寸大幅跳动
- 当前分组应有明确标题，避免用户迷失位置

### 即时保存反馈

- 多数开关与模式类配置采用即时生效
- 即时生效后要有轻量反馈，例如控件状态变化、次级说明更新或短暂状态提示
- 不建议让用户频繁点“保存”按钮

### 快捷键相关交互

- 快捷键项应具备“当前状态”“是否启用”“是否冲突”的清晰可见性
- 录入快捷键时，需要进入专门捕获态，而不是普通输入框体验
- 冲突时应在该项就地提示，而不是只在全局报错

### 高级能力入口

- Warp 跟随显示、通知策略、终端集成等项应有独立 section
- 这些 section 需要支持更长说明文案，因为它们比普通开关更复杂
- 第一版即使暂未全部实现，也应在结构上给出明确位置

## 8. 第一版配置项清单

建议第一版设置窗口采用以下一级导航：

- `Appearance`
- `Shortcuts`
- `Display Behavior`
- `Notifications`
- `Storage`
- `Terminal`

### 8.1 Appearance

建议包含：

- `Theme`
  - `System / Light / Dark`
- `Panel Width`
  - 建议替代当前 `Full width` 的命名
  - 可采用 `Compact / Wide`
- `Visual Density`
  - 第一版可预留，不必强制实现

### 8.2 Shortcuts

建议包含：

- `Show / Hide Walkinal`
- `Alternate Shortcut`
  - 如需保留备用快捷键
- `Conflict Detection`
- `Shortcut Status`
- `Reset to Default`

这一组应作为第一版重点区域之一。

### 8.3 Display Behavior

建议包含：

- `Warp Follow Mode`
- `Follow Position`
  - 底部居中 / 底部右侧 / 上次位置
- `Show Behavior`
  - Warp 激活时是否自动显示
- `Hide Behavior`
  - 离开 Warp 时是否自动隐藏
- `Remember Manual Position`

第一版可以先保留结构，后续逐步实现。

### 8.4 Notifications

建议包含：

- `Notification Sound`
- `Send Success Feedback`
- `Send Failure Feedback`
- `Run Completion Feedback`
- `Only notify when hidden`

### 8.5 Storage

建议包含：

- `Storage Folder`
- `Open Storage Folder`
- `History Retention`
  - 第一版可预留
- `Clear Local History`
  - 第一版可预留
- `Draft Recovery`
  - 后续可扩展

### 8.6 Terminal

建议包含：

- `Terminal Target`
  - 当前先显示 `Warp`
- `Send Behavior`
  - 为未来策略配置预留
- `Open in Terminal`
  - 行为说明或配置位
- `Future Adapter Support`
  - 可作为第一版占位区域

## 8.7 第一版建议真正落地的条目

第一版建议真正实现：

- `Theme`
- `Panel Width`
- `Show / Hide Shortcut`
- `Shortcut Conflict Status`
- `Notification Sound`
- `Storage Folder`
- `Warp Follow Mode` 预留位
- `Terminal Target` 展示或基础配置位

## 8.8 第一版不建议放入的内容

第一版不建议加入：

- 账号系统
- 云同步
- 高级诊断
- 实验功能市场
- 过细的 UI 偏好
- 没有明确实现路径的占位设置

## 9. 与现有 popover 的关系

建议不要直接删除当前 popover 的全部能力，而是做分流：

### 保留在主界面的能力

保留：

- `History`
- 进入设置窗口的入口

### 移入设置中心的能力

当前 popover 中这些项应迁入设置窗口：

- `Full width`
- `Notification sound`
- `Storage folder`
- `Dark theme`

后续也应放入设置窗口的能力包括：

- 快捷键
- 显示行为
- Warp 跟随显示
- 通知策略
- 终端集成配置

## 10. 第一版范围

第一版建议只做结构和承载，不追求一次性把所有未来能力都实现。

第一版目标：

- 有独立设置窗口框架
- 有清晰导航和内容分组
- 能承接当前 popover 中已有设置项
- `History` 继续留在主界面，不迁入设置中心
- 为快捷键、Warp 跟随显示等功能预留位置

第一版不必强求：

- 所有未来功能都立即实现
- 复杂动画和视觉细节一次到位
- 完整账户体系或外部集成页一次性齐备

## 11. 分阶段任务拆分

### Phase 1：方案与结构

- 明确独立设置窗口的入口、退出和整体结构
- 明确一级导航分组
- 明确 `History` 留在主界面、当前 popover 设置迁入设置窗口
- 明确保存策略和反馈方式

### Phase 2：信息架构落地

- 搭建独立设置窗口骨架
- 落定导航与内容区结构
- 接入当前已有设置项
- 替换当前 popover 的主体功能，保留入口但迁移内容

### Phase 3：扩展承载

- 接入快捷键自定义区域占位
- 接入 Warp 跟随显示区域占位
- 为通知与终端集成预留扩展位

### Phase 4：视觉精修

- 提升分组层次、说明文案和排版质量
- 优化控件一致性
- 提升整体质感，使其接近成熟桌面伴随工具

## 11.1 Phase 执行约定

从这份方案开始落地时，可以直接使用：

- `设置中心重构 Phase 1`
- `设置中心重构 Phase 2`
- `设置中心重构 Phase 3`
- `设置中心重构 Phase 4`

作为实现指令。

默认含义如下：

- `Phase 1`
  只做独立设置窗口骨架、入口、退出和导航结构，不迁移现有设置项
- `Phase 2`
  迁移当前 popover 中已有设置项进入新设置窗口，不扩展未来功能
- `Phase 3`
  为快捷键、Warp 跟随显示、通知策略、终端集成等能力预留承载位
- `Phase 4`
  做视觉精修、排版统一和桌面级质感提升

如果没有额外说明，按以下规则执行：

- 只做当前 Phase
- 不提前实现后续 Phase 内容
- 不顺手扩大到未明确批准的额外功能

## 12. 风险与边界

主要风险：

- 如果先做视觉、不先做架构，后续仍然会堆乱
- 独立窗口会引入新的窗口生命周期和焦点管理问题
- 如果主界面和设置窗口边界不清，会造成体验割裂

边界：

- 本方案不直接规定具体像素、视觉稿或组件实现细节
- 本方案不直接展开快捷键功能和 Warp 跟随功能的实现方案
- 本方案只负责定义设置中心如何承载这些能力

## 13. 建议结论

建议按以下原则推进：

1. 先做独立设置窗口框架，而不是继续扩展当前 popover
2. 先做信息架构和交互边界，再做视觉精修
3. 第一版优先承载当前 popover 中已有设置 + 为未来功能预留位置
4. 快捷键和 Warp 跟随显示应视为设置中心重构的第一批受益功能

## 14. 实施 Checklist

以下 checklist 作为后续实现时的执行顺序参考。

### Step 1：建立独立设置窗口

- [ ] 明确设置窗口是独立 `BrowserWindow` 还是当前窗口中的独立 route/view 容器
- [ ] 确定设置窗口的打开方式、关闭方式和聚焦行为
- [ ] 确定设置窗口尺寸、最小尺寸和多屏行为
- [ ] 明确主悬浮层打开设置后是否保持可见

### Step 2：迁移现有设置能力

- [ ] 将当前 popover 中的 `Full width` 迁入设置窗口
- [ ] 将当前 popover 中的 `Notification sound` 迁入设置窗口
- [ ] 将当前 popover 中的 `Storage folder` 迁入设置窗口
- [ ] 将当前 popover 中的 `Dark theme` 迁入设置窗口
- [ ] 保留顶部设置入口，但改为“打开设置窗口”

### Step 3：建立信息架构

- [ ] 搭建左侧导航结构
- [ ] 搭建右侧内容区结构
- [ ] 为一级分组建立稳定骨架：外观、快捷键、显示行为、通知、存储、终端集成
- [ ] 为每个分组补充标题、副说明和内容容器

### Step 4：明确状态与持久化

- [ ] 明确所有设置项的即时保存策略
- [ ] 明确保存成功与失败的反馈方式
- [ ] 确保迁移后不改变现有设置项的实际存储语义
- [ ] 明确未来快捷键、跟随显示等配置项的状态归属

### Step 5：为后续功能预留承载位

- [ ] 为快捷键自定义与冲突检测预留独立 section
- [ ] 为 Warp 跟随显示预留独立 section
- [ ] 为通知策略扩展预留独立 section
- [ ] 为终端集成与 adapter 扩展预留独立 section

### Step 6：视觉精修

- [ ] 优化导航栏层次、间距和选中状态
- [ ] 优化内容区的模块边界和留白节奏
- [ ] 统一控件尺寸、字体层级和说明文案风格
- [ ] 确保整体质感达到桌面级设置中心预期

### Step 7：验收标准

- [ ] 用户可稳定进入和退出设置窗口
- [ ] 当前已有设置项全部可在新设置窗口中完成修改
- [ ] `History` 仍保留在主界面，不迁入设置中心
- [ ] 新设置窗口不会破坏主悬浮层的核心工作流
- [ ] 后续快捷键与 Warp 跟随显示功能可以无结构性重做地接入

## 15. 已确认决策

以下决策已确认，可作为后续实现前提。

### 15.1 窗口形态

- 设置中心采用**真独立窗口**

### 15.2 设置入口

- 顶部 `...` 直接变成“打开设置窗口”的入口
- 不再保留当前设置 popover 主体

### 15.3 第一版范围

- 第一版配置范围按当前方案文档中的建议执行
- 不额外扩大到账号、云同步、高级诊断等方向

### 15.4 状态归属

- 设置中心中的配置统一归入 **app config**

### 15.5 设置窗口与主悬浮层关系

- 打开设置窗口时，主悬浮层保持原状态，不强制隐藏
- 设置窗口独立操作
- 关闭设置窗口不自动改变主悬浮层状态

## 16. 设置反馈策略建议

建议采用**即时保存 + 分级反馈**的方式。

### 16.1 默认策略

- 大多数设置项采用即时保存
- 不额外引入统一“保存”按钮
- 用户修改后应尽快看到状态变化或界面变化

### 16.2 分级反馈

#### A 级：仅控件反馈

适用于低风险、纯 UI 类设置：

- Theme
- Panel Width
- Notification Sound

这些项只需通过控件状态变化和界面即时变化反馈，不需要额外 toast。

#### B 级：就地反馈

适用于有状态、可能失败但不必全局打断的设置：

- 快捷键冲突
- Warp 跟随显示不可用
- Terminal Target 暂不可改
- 某些功能的未启用状态

建议通过：

- 行内状态文案
- 小图标
- 辅助说明
- 当前项下方的错误提示

来表达结果。

#### C 级：全局反馈

适用于有明显副作用或需要强确认的设置：

- Storage Folder 修改成功或失败
- 快捷键注册成功或失败
- 清理历史
- 重置配置
- 需要系统权限但失败

这类项可使用 toast 或更明确的顶部状态提示。

### 16.3 失败提示原则

- 失败优先在设置项附近解释原因
- 避免只给模糊的全局报错
- 如果设置项依赖系统权限、文件系统或窗口能力，应给出下一步建议

### 16.4 建议结论

- 普通开关类设置：即时保存，仅通过控件与界面变化反馈
- 中风险设置：优先行内反馈
- 高影响设置：给明确成功/失败提示，必要时使用 toast
- 反馈要尽量贴近设置项本身，而不是全部上升到全局提示
