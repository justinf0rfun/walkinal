## 输入迁移与习惯过渡

这个方向单独处理，不与一般功能 backlog 混排。

问题本质：

- 用户天然习惯直接在终端原始输入区输入
- 如果 Walkinal 只是“另一个输入框”，用户会持续被旧习惯拉回终端
- 真正要解决的不是单个交互细节，而是“什么场景下用户会主动切到 Walkinal”

目标：

- 不是强迫用户放弃终端输入
- 而是让用户在高价值输入场景下，主动觉得 Walkinal 更划算
- 把 Walkinal 定位成终端输入增强层，而不是所有输入的替代品

需要深想和验证的点：

- [ ] 明确定义 Walkinal 优先使用的高价值输入场景，例如多段整理、附件发送、模板化输入、历史复用
- [ ] 强化 Walkinal 相比终端原始输入区的核心差异能力，确保迁移有明确收益
- [ ] 设计“进入复杂输入模式”的最短路径，让 Walkinal 更像终端增强层而不是独立替代输入框
- [ ] 建立清晰的使用分工，让用户形成“简单输入在终端，复杂输入在 Walkinal”的稳定心智
- [ ] 用真实使用路径验证用户是否会在高价值场景下主动切到 Walkinal
- [ ] 明确这一方向的成功标准，例如使用频率、复杂输入场景切换率、发送前整理使用率

## 备注

- 当前进度以代码实现为准。
- 历史链路已切到 `history-index.json` + 索引查询，`HistoryPicker` 已支持搜索和模式过滤。
- 仓库内部仍保留 `window.clui`、`clui:*` IPC 前缀等兼容性命名，但文档已改按 Walkinal 现状描述。
- 下一阶段应优先做“发送链路稳定性”和“产品收口”，再推进多终端与测试层。
- 当前还存在一类高优先级队列问题：多个队列项发送到 Warp 时需要保留明确段落边界，避免被拼成单行。
- `/` 触发快捷操作的需求参考了 `Claude Code` 的 slash commands 和 `Codex CLI` 的 slash-command 交互预期，应按输入首字符触发的命令面板来设计，而不是普通文本建议。
- 对于“提交代码”“继续”“确认”这类高频短指令，更适合走瞬时发送模式，不进入常规队列和历史，避免污染记录。
- 高 ROI 功能优先补在 `P1`：发送前预览、失败后恢复、历史二次利用、tab 上下文配置、轻量发送反馈。
